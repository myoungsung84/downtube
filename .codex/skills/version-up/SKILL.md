# Version Up

프로젝트 버전을 한 번에 올릴 때 사용한다.

## 범위

- 1차 수정 대상은 아래 3개 파일만 수정한다.
- `package.json`
- `README.md`
- `README.ko.md`

## 지원 시나리오

### `$version-up`

1. `package.json`의 현재 `version` 값을 읽는다.
2. semver 기준으로 patch를 1 올린 목표 버전을 계산한다.
3. 계산한 목표 버전으로 바로 수정한다.
4. 중간에 사용자에게 다시 묻지 않는다.

예:

- `2.1.0` -> `2.1.1`

### `$version-up 2.2.0`

1. 사용자가 직접 입력한 버전을 목표 버전으로 사용한다.
2. 입력값이 semver 형식인지 먼저 확인한다.
3. 유효하면 바로 수정한다.
4. 중간에 사용자에게 다시 묻지 않는다.

허용 예:

- `1.0.12`
- `2.1.0`
- `2.0.0-beta.1`

## 작업 절차

1. `package.json`에서 현재 버전(`<old>`)을 읽는다.
2. 입력 버전이 없으면 `<old>` 기준 patch +1로 `<new>`를 계산한다.
3. 입력 버전이 있으면 semver 형식 검증 후 `<new>`로 사용한다.
4. `package.json`의 `version` 값을 `<new>`로 바꾼다.
5. `README.md`에서 현재 릴리즈 버전을 나타내는 표기만 `<new>`로 바꾼다.
6. `README.ko.md`도 같은 기준으로 바꾼다.

## README 수정 규칙

- 바꿔야 하는 대상은 현재 릴리즈 버전이라고 명확히 판단되는 표기만이다.
- 우선 확인 대상:
- `release badge`
- `Release` 텍스트
- 기타 현재 릴리즈 버전이라고 명확히 보이는 표기
- 관련 없는 숫자, 라이브러리 버전, 예시 숫자, 문서 내 다른 수치는 바꾸지 않는다.
- 예상한 버전 표기를 찾지 못하면 무리하게 넓은 범위 치환을 하지 않는다.
- 찾지 못한 항목은 마지막 보고의 확인 사항에 남긴다.

## semver 규칙

- 직접 입력 버전은 semver 형식만 허용한다.
- semver가 아니면 수정하지 않는다.
- 이 경우 왜 중단했는지 짧게 보고한다.

## 출력 규칙

- 수정 후 최종 보고는 아래 형식을 그대로 사용한다.
- `<old>`, `<new>`만 실제 값으로 바꾼다.
- 찾지 못한 표기나 수동 확인 필요 항목이 없으면 `없음`으로 적는다.

## 사용 예시

```text
$version-up
$version-up 2.2.0
```

```text
[done]

1. 수정한 파일
- package.json
- README.md
- README.ko.md

2. 변경 내용
- version: <old> -> <new>
- README.md 버전 표기 반영
- README.ko.md 버전 표기 반영

3. 확인 사항
- 찾지 못한 표기 또는 수동 확인 필요 여부
```
